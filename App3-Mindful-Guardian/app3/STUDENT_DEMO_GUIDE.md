# 學生操作與報告講解稿：AI 校園心靈守護者

## 一句話介紹

我們這組做的是「匿名、非診斷的 AI 校園指揮中心」。它不是判斷誰有病，而是用亮色產品級儀表板主動巡查校園空間訊號，幫老師更早看到需要關心的區域，必要時可以派遣機器人做提醒與引導。

## 學生操作入口

- 公開展示網址：`https://timdirty.github.io/115-campus-ai-demo/app3/`
- 本機開發版：執行 `npm run dev` 後開啟 Vite 顯示的網址。
- 沒有 Firebase、Gemini 或 Arduino 時也能完整操作，資料會存在瀏覽器本機資料。

## 上台分工建議

- 第一位同學：介紹為什麼校園需要早期關懷。
- 第二位同學：操作校園空間指揮中心與機器人派遣。
- 第三位同學：打開預警與感知抽屜，說明 AI 主動巡查。
- 第四位同學：展示自我照護、聊天支持、硬體紀錄、匯出或重置。

## 第一屏導覽講解稿

大家好，我們是 AI 校園心靈守護者小組。我們的作品不是心理診斷工具，也不收集真實學生個資，而是用匿名資料協助老師更早發現需要關心的情況。

首先在第一屏，可以看到一張 2.5D 校園空間指揮地圖。每個區域都有風險、聲量和提醒數，這些資料都是匿名示範訊號，重點是呈現「AI 主動巡查、非診斷、可持續追蹤」。

如果某個區域變成中高風險，我們可以直接點選它，指派機器人前往介入。這裡的機器人不是取代老師，而是先做柔和提示、引導或通知，接著由老師低壓確認。

接著打開預警抽屜。我們選一筆提醒，勾選處置清單，例如「導師已確認」、「安排關懷談話」或「轉介輔導室」。按下佈署關懷後，系統會新增支持方案，也會留下硬體提示紀錄。

再打開感知抽屜。系統可以用本機麥克風只計算音量和波動，不保存錄音，也不轉文字。AI 會融合心情簽到、聲量、節點和提醒，主動判斷是否需要老師觀察。

最後打開照護與紀錄抽屜。學生可以做心情簽到、匿名心情牆與安全空間聊天；紀錄抽屜則能看到機器人任務和硬體提示，證明整個流程有留下可追蹤證據。

## 操作步驟卡（底部 4 個 Tab：預警 / 感知 / 照護 / 機器人）

1. 開啟第一屏，先說明 AI 主動巡查校園空間。
2. 指出最高風險區的風險、聲量與提醒數。
3. 點選中高風險區域，指派機器人介入。
4. 點 **預警** Tab → 選一筆提醒並勾選處置清單。
5. 點 **感知** Tab → 按「示範高聲量」或啟用麥克風，看聲量趨勢圖；按「建立提醒」。
6. 點 **照護** Tab → 做一次心情簽到，發一篇心靈森林貼文，或聊天。
7. 點 **機器人** Tab → 確認節點狀態、硬體提示紀錄與支持方案。
8. 最後展示匯出、匯入或重置（Header 右上角按鈕）。

## 評審可能會問

### 這是不是在診斷學生心理狀態？

不是。我們只做關懷提醒和支持建議，不做醫療或心理診斷，也不使用真實學生姓名。

### 為什麼要匿名？

因為心理與情緒資料很敏感。展示時用匿名代號，正式使用也應該先保護隱私，再讓老師或輔導室提供支持。

### 沒有 Firebase、Gemini 或 Arduino 可以展示嗎？

可以。系統目前是 local-first，沒有雲端也能完整操作；沒有 Arduino 時會保留硬體提示紀錄，接上 UNO R4 後同一操作會送到實體提示裝置。

### 機器人出現會不會標籤化學生？

機器人是輔助老師「去關心」，不是去「標記誰有問題」。它會在高風險空間附近做柔和音效或燈號提示，不會貼近特定學生、不會廣播姓名，也不會顯示任何個人資訊。老師看到提示後再用低壓方式確認，決定權永遠在人，不在機器。

### 你們的 AI 只是 if-else 規則，還是真的在做 AI？

我們有三層真正的 AI 融合：① `proactiveGuardian.ts` 把心情簽到分佈、聲量指數、節點狀態、未解提醒四個訊號加權合成一個風險分數，不是單一規則；② `acousticGuardian.ts` 即時分析麥克風波形算音量指數與波動性，而不是只判斷大小聲；③ `localGuardianAi.ts` / Gemini API 根據提醒上下文生成個人化支持建議。這三層結合才觸發派遣決策，比單純 if-else 複雜很多。

## 後續機器人連動計畫

1. 將 `ALERT_SIGNAL` 對應到柔和 LED 提醒，避免造成學生壓力。
2. 將 `CARE_DEPLOYED` 對應到關懷已佈署的提示圖示。
3. 將 `NODE_RESTART` 對應到節點重新連線燈號。
4. 保留 `NODE_HEARTBEAT` 給未來節點定期回報。
5. 堅持不在硬體端顯示真實學生姓名或敏感內容。
